# `analysis/configs/` — Configuration Canon

The configs directory holds **canonical base configs** at the root and
**scenario-specific overrides** under `variants/`. The split exists so that
the small set of files at the root is the "single source of truth" and
everything under `variants/` is an explicit override of one of them via
`_base:` inheritance.

## Canon definition

A config is **canonical** if **either** of the following is true:

1. It has no `_base:` key (it is a true root in the inheritance graph), OR
2. It is loaded directly by `ced_ml` CLI commands as a default
   (e.g. `consensus_panel.yaml`, `optimize_panel.yaml`, `permutation_test.yaml`,
   `aggregate_config.yaml`, `holdout_config.yaml`, `output_config.yaml`).

The 10 canon files are referenced by name from inside `ced_ml/cli/` —
**do not rename or move them** without updating the corresponding default
paths in `commands/aggregation.py`, `commands/evaluation.py`,
`config/loader.py`, `cli/run_pipeline.py`, `cli/hpc/submission.py`, and
`cli/utils/config_merge.py`.

## Variants

A **variant** is a config that declares `_base: <path>` and inherits the
referenced config, then deep-merges its own values on top. Variants in
this tree always inherit from a canon file or from a sibling variant.
Each variant lives under `variants/<scenario>/`.

`_base:` paths are **resolved relative to the file containing the
declaration** (see `ced_ml/config/loader.py:65`). After the move into
`variants/<scenario>/`, paths like `_base: training_config.yaml` were
rewritten to `_base: ../../training_config.yaml`. Sibling references
(e.g. `training_config_val_consensus_100t.yaml` → `training_config_val_consensus.yaml`)
stayed unchanged because both files are in the same directory.

|---------------------------------|------------------------------------------|-------|
| `analysis/configs/recipes/`     | `results/_archive/gen1/derived_recipes/` | 448 files. Output of `ced derive-recipes` from before the manifest move; the new generation target is `operations/cellml/configs/recipes/`. |

Both archives are gitignored (under `results/`); the moves are recorded in
git as deletions of the old paths. If you ever need to read them, the
files still exist on disk under the archive root.

## Adding a new variant

1. Pick the scenario it belongs to (`val_consensus/`, `holdout/`, `4protein/`,
   `local/`) or create a new subdirectory if your scenario is genuinely new.
2. Set `_base: ../../<canon>.yaml` (or `_base: <sibling>.yaml` if extending
   another variant in the same dir).
3. Override only the keys that differ from the base. Keep variants minimal —
   if you find yourself overriding most of the canon, the canon is wrong.
4. Run `python -c "from ced_ml.config.loader import load_yaml; load_yaml('analysis/configs/variants/<scenario>/<file>.yaml')"`
   to verify the chain resolves.

## Adding a new canon config

Generally avoid this. The canon is intentionally small. If you must:

1. Add the file at the root of `analysis/configs/`.
2. Wire its default path into the appropriate CLI command.
3. Update this README.
4. Update `CLAUDE.md` and `docs/architecture.md`.
